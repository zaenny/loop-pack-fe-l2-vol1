# /self-review — AI 코드 검증 스킬

AI가 짠 코드에서 사람이 놓치기 쉬운 함정을, 커밋·PR 전에 기계적으로 잡는 백스톱.
사람이 직접 diff를 읽는 규율을 대체하지 않는다 — 그 위에 덧대는 검증이다.

각 항목은 **추측하지 말고 실제 파일/명령으로 확인**한다. (AI는 그럴듯한 거짓 보고를 하기 쉽다)
**근거 없는 발견을 만들지 않는다.** 줄 번호를 지어내지 않고, 실제로 확인한 위치만 보고한다. 확실하지 않으면 "확인 필요"로 표기한다.

---

## 절차

1. `git diff`로 변경 파일 목록을 확인하고, 변경된 모든 파일을 Read로 읽는다.
2. `CLAUDE.md`를 읽어 규칙을 확인한다.
3. 아래 관점을 검사하고 결과를 형식에 맞춰 보고한다.

---

## 검사 항목

### 1) 의존성 — 실존 확인

- import한 패키지가 실제로 존재하는가. → `package.json`의 dependencies에 있는지, 또는 `node_modules`에 있는지 **직접 확인**한다. 기억으로 판단하지 않는다.
- import한 함수/컴포넌트가 해당 모듈에 실제로 export되어 있는가. → 대상 파일을 열어 확인한다.
- 기존 유틸을 두고 같은 기능을 새로 만들지 않았나. → 비슷한 이름·기능을 코드베이스에서 검색해 중복을 찾는다.
- hook 의존성 배열이 완전한가. (`useEffect`/`useMemo`/`useCallback`에서 참조하는 값이 deps에 다 들어있는지)

### 2) 침묵된 경고 — "기계가 경고하려던 자리"

변경된 diff에서 아래를 grep으로 찾는다. 하나라도 새로 추가됐으면 그 자체로 지적한다.

- `any` (명시적 타입 회피)
- `as` (타입 단언 — 특히 `as any`, `as unknown as`)
- `@ts-ignore`, `@ts-expect-error`
- `eslint-disable`, `eslint-disable-next-line`

각각은 "도구가 경고하려다 막힌 자리"다. 정당한 사유가 주석으로 없으면 `blocking` 또는 `important`로 보고한다.

### 3) XSS 표면적

- `dangerouslySetInnerHTML` 사용처가 있는가. 있으면 입력이 신뢰 가능한지/sanitize됐는지 확인한다.
- 외부·사용자 입력을 검증 없이 그대로 렌더링하거나 URL/속성에 넣는 곳이 없는가.

### 4) 처리 누락 (구조적 탐지만)

happy path만 처리하고 빠진 분기가 없는지 **코드 구조 기준으로** 본다. (실제 동작 시뮬레이션은 사람 몫)

- 데이터를 다루는 컴포넌트/훅에 로딩 · 에러 · 빈 상태 처리가 있는가.
- 상태 전이의 반대 방향이 처리됐나. (있다→없다, 성공→실패)
- early return 없이 정상 경로만 가정하고 있지 않은가.
- 빈 catch 블록(`catch (e) {}`)이 없는가. 에러를 잡아놓고 아무것도 안 하면 에러를 삼키는 것이다.

### 5) React 안티패턴

- **파생값을 state로 동기화하지 않는가 (최우선).** 다른 state나 props로 계산할 수 있는 값을 `useState` + `useEffect`로 베껴 넣고 있지 않은지 본다. 파생 가능한 값은 렌더 중 계산해야 한다. (AI 코드 단골 실수 — 발견되면 `blocking`/`important`)
- **early return이 모든 hook 호출보다 뒤에 있는가.** hook 호출보다 먼저 `return`하면 rules of hooks 위반이라 런타임 버그가 된다. (발견되면 `blocking`)

## 결과 형식

```
## Self Review 결과

검토 파일: {N}개 · 발견: {M}건
판정: APPROVE / REQUEST CHANGES / COMMENT
(blocking이 하나라도 있으면 REQUEST CHANGES)

### blocking
- {파일}:{줄} — {문제} → {고칠 방법}

### important
- {파일}:{줄} — {문제} → {고칠 방법}

### nit
- {파일}:{줄} — {문제} → {고칠 방법}

### suggestion
- {파일}:{줄} — {문제} → {고칠 방법}
```

- 해당 심각도에 발견이 없으면 그 섹션은 생략한다.
- 모든 항목은 실제로 확인한 `파일:줄`만 적는다. 위치가 불명확하면 줄 번호 대신 "확인 필요"로 적는다.

심각도 기준:

- `blocking` — 머지하면 안 되는 것 (버그, 보안, 존재하지 않는 import, 정당한 사유 없는 타입/린트 침묵, hook 규칙 위반)
- `important` — 머지 전 고치는 게 맞는 것 (처리 누락, 중복 구현, 파생값 state 동기화)
- `nit` — 사소한 개선
- `suggestion` — 선택적 제안

---

수정 정책
코드를 자동으로 고치지 않는다. self-review는 진단만 한다.
각 발견에 대해 "무엇이 문제인지"와 "어떻게 고치면 좋은지"를 결과 형식의 → {고칠 방법}에 구체적으로 적는다. (가능하면 수정 예시 코드를 함께 제시한다.)
실제 수정 여부와 방법은 사용자가 직접 결정한다. 사용자가 명시적으로 "고쳐줘"라고 요청하기 전까지 파일을 수정하지 않는다.
