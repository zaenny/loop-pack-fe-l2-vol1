## 🎯 왜 이 과제를 하는가

- **AI가 쏟아내는 코드의 결함은 대부분 "한 파일에 모든 책임"이다.** 동작은 하지만 변경 비용이 폭증한다. 정렬 조건 하나 바꾸려고 500줄을 읽게 되는 순간, "잘 돌아가는 코드"가 "고칠 수 없는 코드"가 된다.
- **레이어를 가른다는 건 추상화가 아니라 "한 번에 한 가지만 읽고 싶다"는 욕구다.** UI 코드만 봐도 화면이 그려지고, Hook만 봐도 기능 흐름이 잡히고, Service만 봐도 API 스펙이 보여야 한다.
- **분리 자체가 목적이 아니다 — "이 분리가 무엇을 좋게 하나"를 한 문장으로 답할 수 있어야 한다.** 답이 없는데 분리하면 "큰 컴포넌트"가 "큰 Hook"으로 옮겨갈 뿐이다. 이번 주 과제의 절반은 **고치지 않겠다는 결정**이다.
- **상태 3분할(서버 / 클라이언트 / 파생값) 감각이 모든 결정의 기반이 된다.** "Custom Hook이 뭐고 상태관리가 뭐예요?"라는 혼란은 대부분 이 세 가지를 구분하지 않아서 생긴다.

## 🗺️ 어떻게 진행하나

각 항목의 자세한 설명은 강의 자료를 참고한다.

- **3개 레이어** — Components(UI) / Hooks(로직) / Services(API). 순수 함수는 Utils.
- **Custom Hook은 한 문장으로 설명된다**
- **상태 3분할** — 서버 / 클라이언트 / 파생값을 구분한다.
- **useEffect는 외부 시스템 동기화 도구**
- **DIP** — Hook은 `fetch`·`axios` 구현체에 직접 묶이지 않고 service 함수에만 의존한다.

## 👀 셀프 리뷰 (리뷰 요청 전, 내 diff를 처음부터 끝까지)

1. **레이어** — 컴포넌트 파일만 봐도 화면이 그려지고, Hook 파일만 봐도 기능 흐름이 잡히고, Service 파일만 봐도 API 스펙이 잡히나?
2. **Hook 책임** — 각 Custom Hook을 **한 문장**으로 설명할 수 있나? 한 문장에 "그리고"가 두 번 들어가면 분리 후보다.
3. **상태 3분할** — 서버 상태·클라이언트 상태·파생값을 의식적으로 구분했나? 파생값을 state로 들고 있지 않나?
4. **Effect 정당성** — 남은 `useEffect`마다 "외부 시스템과의 동기화인가"라는 질문에 답할 수 있나?
5. **DIP** — Hook이 `axios`·`fetch` 구현에 직접 묶여 있지 않나? 통신 구현이 service에 모여 있나?
6. **분리하지 않은 결정** — "여긴 분리 안 한다"는 결정에도 한 줄 근거가 있나?
