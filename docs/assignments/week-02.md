# 2주차 — 컴포넌트 & Props 전략: Bad smell을 찾아 리팩터링 하기

> 체크아웃 화면에서 **Bad Code를 판별하고 리팩토링한다.**

## 왜 이 과제를 하는가

- **판별력이 가장 자주 필요한 지점이 props 설계다.** AI는 "돌아가는 컴포넌트"를 순식간에 쏟아낸다. 진짜 병목은 _그게 잘 설계됐는지 읽어내는 눈_ 이고, 그 눈이 가장 먼저 드러나는 곳이 경계·계약·합성이다.
- **제1원칙의 2주차 변형 — 근거 없는 리팩토링은 커밋하지 않는다.** 1주차 원칙("설명할 수 없는 코드는 커밋하지 않는다")은 고칠 때도 똑같이 적용된다. "왜 나눴나 / 왜 합쳤나 / 왜 옮겼나"를 **한 문장**으로 답하지 못하면 되돌린다.
- **과잉 리팩토링도 냄새다.** 필요 없는데 추상화하고, 굳이 상태를 위로 끌어올리는 건 "부지런한 실수"다. 이번 과제의 절반은 **고치지 않겠다는 결정**이다. 강의에서 던진 메타 질문 — _"지금 추상화할 때인가?"_ — 을 과제 내내 곁에 둔다.

## 이번 주 과제

스타터는 `src/market/` 체크아웃(주문/결제) 화면이다. **동작은 한다.** 할 일은 기능 추가가 아니라 **판별 → 리팩토링 → 근거 기록**이다.

| #   | 과제                                          | 완료 기준                                                                                                |
| --- | --------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| 1   | **냄새 판별** — 강의 11전략으로 코드를 훑는다 | `위치 · 전략 · 왜 냄새인가 · 고칠 방향` 표 1개. **"해당 없음"으로 판단한 전략도 1줄 근거와 함께** 적는다 |
| 2   | **리팩토링** — 판별한 냄새만 고친다           | **동작 보존** + 셀프 리뷰 4단계 통과                                                                     |
| 3   | **PR** — 1주차와 같은 흐름                    | AI 생성 부분 표기 + **리팩토링마다 한 줄 근거**                                                          |

## 어떻게 진행하나

강의에서 다룬 **11개 전략이 그대로 체크리스트**다. `src/market/`의 **모든 파일**을 이 렌즈로 한 번씩 훑으면서, 각 전략이 _여기 적용되는지, 해당 없는지_ 를 스스로 판단한다.

- **경계** — ① 변화의 경계 · ② 구현 vs 조합 · ③ God Component · ④ 성급한 추상화
- **계약** — ⑤ props는 적게, 이름은 역할대로 · ⑥ boolean 폭발 → enum · ⑦ 파생 상태 + key · ⑧ 확장은 위임으로
- **합성** — ⑨ Context 전에 composition · ⑩ children vs slot · ⑪ Drilling vs Context

> 코드만 읽지 말고 **화면을 직접 조작**해보자. 눈으로 안 보이던 냄새가 동작에서 드러난다.
> 그리고 **11개가 다 들어 있는 건 아니다.** "여긴 해당 없음"이라는 판단도 점수다.

### 코드를 볼 때 던지는 질문

1. 이건 **몇 가지 이유**로 바뀌나? (하나를 넘으면 경계)
2. **props로 줄까, 자리(합성)를 내줄까?** (동작은 prop, 구조는 합성)
3. 이 **값은 어디 살아야** 하나? 파생값이면 **계산**했나?

## 🚫 이렇게는 하지 마세요

- **냄새가 아닌데 "고쳤다" 하지 않기.** 필요 없는 추상화, 과한 상태 끌어올리기가 대표적이다. **과잉 리팩토링도 제1원칙 위반**(설명할 수 없는 변경)이다.
- **강의 예제 해법을 그대로 복붙하지 않기.** 슬라이드 코드를 옮기면 전이가 아니라 암기다. 이 코드에 맞게 스스로 다시 푼다.
- **동작을 바꾸지 않기.** 리팩토링은 동작 보존이 원칙이다. 다만 냄새가 가려놓은 버그가 드러나 고쳐지는 건 동작 변경이 아니라 _복구_ 다.
- **타입·린트를 침묵으로 때우지 않기.** 새 `any`·`as`·`@ts-ignore`·`eslint-disable`로 누르는 순간, 거기가 바로 린터가 경고하려던 자리다.

## 셀프 리뷰 (리뷰 요청 전, 내 diff를 처음부터 끝까지)

1. **동작** — happy path 말고 **상태 전이**(적용→해제, 선택 변경)까지 고려한 변경인가?
2. **의존성** — hook 의존성 배열은 완전한가? 이미 있는 타입·유틸을 두고 새로 만들지 않았나
3. **타입·린트 침묵** — 새 `any`·`as`·`disable`은 없나
4. **표면적** — 입력값을 검증 없이 렌더링하는 구멍은 없나

→ PR 본문에 한 줄: _"○○는 AI로 생성한 뒤 직접 검토·수정했습니다."_

## 좋은 코드 기준 (10주 공통)

- 의도가 이름에 드러난다 (`data`·`temp`·`flag` ✗, 공통 컴포넌트에 도메인 용어 ✗)
- 한 컴포넌트는 한 가지 일만 한다 / props가 5개를 넘으면 재검토
- **파생 가능한 값은 계산한다** — 상태로 복사해 동기화하지 않는다
- Context보다 composition을 먼저 / 타입이 코드를 설명한다 (`any`로 회피 ✗)
- 이미 있는 코드를 재사용한다 (비슷한 걸 매번 새로 만드는 게 AI의 고질병)
